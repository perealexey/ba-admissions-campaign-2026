# Пайплайн приёмной кампании бакалавриата ВШЭ — как это устроено

Документ исторической ценности: описывает **действующий** процесс сбора и
анализа данных, каким он сложился к моменту первого написания (после
разведки, сбора, метрик, дашборда и первого деплоя). Не хроника ошибок и
доработок — только то, что реально работает и как этим пользоваться в
следующий раз (например, в следующую приёмную кампанию).

Формулы и обоснование метрик — отдельно, в [`METRICS.md`](METRICS.md). Разбор
находок по структуре данных (программы-близнецы, целевая квота, приоритеты) —
в [`FINDINGS.md`](FINDINGS.md). Здесь — только то, как данные превращаются из
сырого JSON API в готовые таблицы и дашборд.

---

## 0. Рамка проекта

- **Источник**: недокументированный JSON REST API Angular-приложения
  `pk.hse.ru/admissions/bak/BD`. Доступ анонимный, без cookie/токенов.
  `robots.txt` на домене отсутствует технически (проверено живым запросом) —
  формального запрета нет, но это не то же самое, что организационное
  согласование; см. разбор в истории проекта, если понадобится пересмотреть.
- **Единственная соблюдаемая мера вежливости** — задержка 2 сек между
  запросами и идентифицирующий `User-Agent` с контактом (в
  `collector_bak.py`). В отличие от `priem47.hse.ru` (магистратура), для
  `pk.hse.ru` не было установлено ограничение «не чаще раза в сутки» — это
  два разных источника с разными договорённостями, не путать.
- **Архитектура**: два независимых контура — (А) сбор+метрики локально,
  (Б) публикация в публичный репозиторий → Streamlit Cloud, данные — через
  приватную папку Google Drive. Оба контура пройдены.
- Данные абитуриентов не публикуются в открытый доступ — ни в репозитории,
  ни где-либо ещё, кроме самого дашборда за паролем.

## 1. Сбор: `collector_bak.py`

**Скрипт**: [`scripts/collector_bak.py`](../scripts/collector_bak.py)

Тянет живой каталог конкурсных групп при каждом запуске (полная иерархия
одним запросом), затем постранично собирает абитуриентов по каждой группе.
Резюмируемый (`bak_data/progress.json`) — повторный запуск не перескачивает
уже собранные группы, безопасно прерывать и перезапускать.

```bash
python3 scripts/collector_bak.py
```

Ключевые решения, зашитые в скрипт (подробный разбор — в `FINDINGS.md`):

- **Целевая квота** дедуплицируется по паре `(setOfCompetitiveGroupId,
  placeTypeId)`, а не по `competitiveGroupId` — иначе один и тот же список
  абитуриентов задваивается по числу заказчиков-организаций. Складывается в
  `bak_data/raw_target_quota/`.
- **Иностранцы — отдельный конкурс** исключены из сбора целиком
  (`SKIP_FOREIGNERS = True`) — решение пользователя, категория целиком
  платная, не пересекается с бюджетным набором. Список пропущенных групп —
  `bak_data/skipped_foreigners.csv`.
- Обычные группы (бюджет, отдельная квота, особое право, платное) —
  `bak_data/raw/{competitiveGroupId}.json`.

Полный прогон (491 обычная группа + 32 пула целевой квоты, задержка 2 сек на
запрос) занимает продолжительное время (десятки минут) — резюмируемость
существует именно для этого, не обязательно проходить его за один присест.

## 2. КЦП (места приёма): `parse_supply_bak.py`

**Скрипт**: [`scripts/parse_supply_bak.py`](../scripts/parse_supply_bak.py)

Отдельный источник — 4 RTF-таблицы мест приёма, по одной на филиал
(`docs/{moscow,nnov,perm,spb}_supply.rtf`), экспортированные с сайта ВШЭ
вручную (не через API — такого источника КЦП в API не нашлось). Структурно
**разные** между филиалами (разное число колонок, разный порядок квот,
Пермь вообще без гиперссылок на программы) — конфигурация под каждый филиал
явно задана в `FILIAL_CONFIGS`, не угадывается по позиции.

```bash
python3 scripts/parse_supply_bak.py docs/ -o data/
```

Встроенная проверка — сумма по строкам сверяется с явной строкой
«Итого»/«Всего» в каждом файле (защита от съехавшего парсинга). Результат —
`data/supply_bak_raw.csv`, разбивка по 6 колонкам квот (бюджет, особое право,
целевая квота, отдельная квота, платные, иностранцы) на каждую программу.

**Меняется редко** — только если ВШЭ обновит места приёма. Не нужно
перезапускать при каждом обновлении данных абитуриентов (см. чек-лист в §5).

## 3. Метрики: `campaign_metrics_bak.py`

**Скрипт**: [`scripts/campaign_metrics_bak.py`](../scripts/campaign_metrics_bak.py)

Читает `bak_data/` (сырые заявки) + `data/supply_bak_raw.csv` (КЦП), считает
M1–M9 (три канала — бюджетная семья/платное/целевая квота, объединённая
желанность через inverse-variance weighting). Полное описание — в
[`METRICS.md`](METRICS.md).

```bash
python3 scripts/campaign_metrics_bak.py bak_data/ data/supply_bak_raw.csv \
    --snapshot-date YYYY-MM-DD -o data/
```

Помимо метрик сохраняет `data/bak_applications_long.parquet` и
`data/bak_budget_family_collapsed.parquet` — сырые данные для дашборда
(вкладка «По абитуриенту», распределения), чтобы не парсить JSON заново на
каждый запуск приложения.

---

## 4. Заливка на Drive: `sync_to_drive_bak.py`

**Скрипт**: [`scripts/sync_to_drive_bak.py`](../scripts/sync_to_drive_bak.py)

```bash
python3 scripts/sync_to_drive_bak.py
```

Заливает всё содержимое `data/` в приватную папку на Google Drive (не в
репозиторий), откуда её читает задеплоенный дашборд. Тот же сервисный
аккаунт, что и в магистратуре (`summer_campaign_26`), но **отдельная**
папка — данные двух кампаний не пересекаются. Подробности инфраструктуры —
§6 ниже.

---

## 5. Сквозной порядок запуска (чек-лист на будущее)

1. `collector_bak.py` → свежие сырые данные (можно не делать при каждом
   мелком изменении кода дашборда — это самый долгий и единственный шаг,
   трогающий сайт ВШЭ).
2. (Опционально, если обновились места приёма) `parse_supply_bak.py`.
3. `campaign_metrics_bak.py` → пересчёт метрик на свежем срезе.
4. `sync_to_drive_bak.py` → залить `data/` на Google Drive.
5. Открыть задеплоенный дашборд, проверить дату среза в шапке (кэш на
   10 минут — см. §6).

В отличие от магистратуры, здесь нет отдельных shell-обёрток
(`fetch_data.sh`/`run_process_and_sync.sh`) — шаги запускаются вручную по
одному, порядок простой и линейный.

## 6. Хостинг: git + Streamlit Cloud + Google Drive

- **Репозиторий** — публичный GitHub (`github.com/perealexey/ba-admissions-campaign-2026`).
  Публичный сознательно, по той же причине, что и в магистратуре: бесплатный
  тариф Streamlit Community Cloud не деплоит из приватных репозиториев. Данные
  абитуриентов в репозитории не хранятся — `data/`, `bak_data/`, `har/`,
  `cowork_output/`, RTF/HTML-источники КЦП — всё в `.gitignore`.
- **Данные** — отдельная приватная папка на Google Drive, доступ только у
  сервисного аккаунта `perealex-drive-reader@theta-sunlight-501810-p1.iam.gserviceaccount.com`
  (тот же аккаунт, что и в магистратуре — переиспользован сознательно, не
  заводили новый проект в Google Cloud; изолирован новой отдельной папкой).
  `app.py` читает файлы из неё через Drive API при каждом запуске (кэш 10 мин).
- **Тема дашборда** (цвета) — `.streamlit/config.toml` в репозитории
  (коммитится, не секрет). Настройка через `--theme.*` флаги командной строки
  (как в локальном `launch.json`) действует только локально — Streamlit Cloud
  их не видит, читает исключительно `config.toml`.
- **Секреты** (пароль дашборда, ключ сервисного аккаунта, ID папки на Drive)
  — не в git. Локально: `.streamlit/secrets.toml`. На Streamlit Cloud:
  Settings → Secrets того же приложения, тот же TOML целиком (включая
  многострочный `private_key` — при вставке в веб-форму сверять, что `\n`
  внутри строки не превращаются в настоящие переносы).
- **Доступ коллегам**: репозиторий публичный на GitHub, дашборд закрыт общим
  паролем (`check_password()` в `app.py`) — пароль раздаётся коллегам, не
  ссылка на GitHub.
