## 1. Источник данных и структура сбора

- Источник — недокументированный JSON REST API Angular-приложения
  `pk.hse.ru/admissions/bak/BD`. Доступ анонимный, без ФИО в данных (только

- Сборщик: [`cowork_output/collector_bak.py`](../cowork_output/collector_bak.py).
  Уже отработан, резюмируемый, с задержкой 2 сек/запрос.
- Результат сбора:
  - `bak_data/raw/*.json` — 405 файлов, один на `competitiveGroupId` (обычные
    группы: бюджет, отдельная квота, особое право, платное).
  - `bak_data/raw_target_quota/*.json` — 32 файла, один на уникальную пару
    `(setOfCompetitiveGroupId, placeTypeId)` целевой квоты (см. §3 — там
    несколько заказчиков делят один список абитуриентов).
  - `bak_data/skipped_foreigners.csv` — 86 групп «Иностранцы — отдельный
    конкурс», исключены из сбора по решению пользователя (см. §4).
  - `bak_data/catalog_snapshots/` — снапшоты полного каталога 566 конкурсных
    групп на момент каждого запуска.

### Поля одной заявки (`applicants[i]`)

```
achievementsSum, achivementsSumTarget, applicantEntranceTests (список
{name:{...,name}, score}), avgCertificateScore, avgMathRussianScore,
contractsCountPerOffer, customerIdRabotaVRossii, customerOfferNumber,

participantStatus, priority, russianLanguageScore, sumCompetitiveScore,
sumCompetitiveScoreTarget, sumEntranceTestScore, withoutExamsAdmReason
```

Ключевые для метрик: `idEpgu` (уникальный человек — аналог `student_id` из
магистратуры), `priority`, `participantStatus` (например «На рассмотрении»,
«Отклонено вузом», «Отозвано поступающим» — статусы заявки, не оценённые ещё
на предмет того, какие считать активными), `sumCompetitiveScore` (итоговый
конкурсный балл), `isConcertToEnrollment` / `isEducationAgreement` /
`isFirstPeriodPayed` (сигналы согласия на зачисление — аналог
`enrollment_consent` из магистратуры).

### Схема файла `raw_target_quota/tq__{setId}__{placeTypeId}.json`

То же самое + два дополнительных поля:
```json
{
  "meta": { /* представитель пула — один из catalog-дублей, любой */ },
  "competitiveGroupIds": ["...", "...", "... (2-8 штук, по числу заказчиков)"],
  "customers": [
    {"competitiveGroupId": "...", "customer": "Второй кассационный суд общей юрисдикции",

```
`customers[].contractsCountPerOffer` — единственный найденный на сегодня
источник числа мест (КЦП) хоть для какого-то трека; для остальных треков
(бюджет, отдельная квота, особое право, платное) источник КЦП **не найден и
не исследован** — это открытый вопрос, см. §6.

## 3. КЛЮЧЕВАЯ НАХОДКА: приоритет общий внутри узкой «бюджетной семьи» —
## но НЕ включает целевую квоту

Статистически проверено (сравнение `priority` одного и того же `idEpgu` на
одну и ту же программу — join по `educationProgramId`, не по названию —
через разные `placeTypeName`):

| Пара `placeTypeName` | Людей с записями в обеих | Priority совпадает |
|---|---:|---:|
| Бюджетные места ↔ Отдельная квота | 5 741 | **99.6 %** |
| Бюджетные места ↔ Особое право | 2 454 | **99.3 %** |
| Бюджетные места ↔ Целевая квота | 260 | **10.4 %** |
| Бюджетные места ↔ С оплатой обучения | 27 252 | 61.2 % |

**Вывод**: «Бюджетные места», «Отдельная квота» и «Особое право» — это НЕ три
независимые конкурсные гонки, а один и тот же приоритет абитуриента на
программу, продублированный по льготным трекам одновременно (претендует на
одно и то же место программы сразу по общему конкурсу и по льготе). Это
узкая **«бюджетная семья»** из ровно этих трёх треков.

**Целевая квота в эту семью НЕ входит** — совпадение приоритетов с бюджетом
всего 10.4%, даже ниже фонового шума «С оплатой обучения» (61%). Это
исправление более ранней ошибки: была высказана промежуточная гипотеза (по
единичному примеру, ещё до сбора целевой квоты), что целевая квота — часть
общей нумерации, и эта гипотеза была ошибочно объявлена подтверждённой без
пересчёта на реальных данных. После сбора всех 32 пулов целевой квоты и
пересчёта выяснилось обратное: у целевой квоты своя, не связанная с бюджетом
нумерация приоритетов — как и у «С оплатой обучения». **Как включать целевую
квоту в итоговые метрики программы — открытый вопрос**, см. §6.1 и
`FINDINGS.md` §5 — не предполагайте автоматически, что её можно слить с
остальной бюджетной семьёй по priority.

«С оплатой обучения» — независимо пронумерованный трек (аналог `commercial`
из магистратуры). Иностранцы исключены из датасета (см. §4).

**Практическое следствие, которое НУЖНО заложить в дизайн метрик**: при
расчёте M1–M9 по программе узкую бюджетную семью (Бюджетные места +
Отдельная квота + Особое право, БЕЗ целевой квоты) нужно **сначала свести к
уникальным людям на программу** (группировка по `idEpgu` + `educationProgramId`),
и только потом считать метрики — иначе один человек посчитается 2–3 раза.
Реальный масштаб этой проблемы небольшой и посчитан точно: из 74 728
уникальных пар (человек, программа) в узкой семье 8 025 встречаются в
нескольких треках одновременно, и лишь у **47** из них `priority`
действительно расходится между треками (0.6% от восьми тысяч, 0.06% от всех).
Выбор правила агрегации (минимальный priority / самый «активный»
`participantStatus` / что-то ещё) меняет результат только для этих 47
человек — их можно и нужно разобрать поимённо при формализации, а не
выбирать правило абстрактно.

## 4. Иностранцы — отдельный конкурс: решение исключить (уже сделано)

Проверено по полному каталогу через управляемое поле `placeTypeCode`/
`placeTypeId` (не по отображаемому названию группы — оно свободный текст и
менее надёжно): все 86 записей категории «Иностранцы — отдельный конкурс»
имеют один и тот же `placeTypeId` с кодом `К` — тот же буквенный код, что и
«С оплатой обучения» (оба «коммерция», разные подвиды по гражданству), и ни
у одной нет кода `Б` (бюджет). Бюджетных мест для иностранных граждан в этом
источнике нет — категория целиком платная, отдельный параллельный трек, не
пересекается с бюджетным набором.

По прямому указанию пользователя категория **исключена из сбора и из
визуализации целиком** — `SKIP_FOREIGNERS = True` в `collector_bak.py`,
данные не попадают в `raw/`. Не нужно предлагать способы её включить или
учитывать в дизайне метрик — только помнить, что общее число
конкурсных групп (566) не совпадает с числом собранных, и это ожидаемо.

## 5. Целевая квота: реальная структура (пробел закрыт)

Изначальная гипотеза (что 75 групп различаются неким скрытым параметром
запроса) была неверной. По HAR с открытой страницы конкретной группы целевой
квоты выяснилось:

- `/applicant?...&setOfCompetitiveGroupId=X&placeType=ЦД` возвращает **один
  общий пул абитуриентов**, одинаковый для всех заказчиков, разделяющих эту
  пару параметров. В самих заявках `customerIdRabotaVRossii`/
  `customerOfferNumber` всегда `null` — привязки заявителя к заказчику там
  нет.
- Реальная привязка живёт в отдельном эндпоинте `/quota?setOfCompetitiveGroupId=
  X&placeTypeId=ЦД` — список организаций-заказчиков с числом договоров
  (`contractsCountPerOffer`) — это про **предложение** (места), не про
  заявителей.
- Из 75 catalog-строк уникальных пар оказалось **32** — это и есть реальное
  число конкурсных пулов. Собраны все 32, без задвоения (проверено: 75 строк
  заказчиков в сумме по всем пулам = 75 исходных catalog-дублей).

Дедупликация API-пулов, описанная выше, установлена надёжно и не
пересматривается. А вот то, что казалось логичным следствием (раз это часть
бюджетного набора мест, то и приоритет должен быть общим с бюджетной семьёй)
— на данных не подтвердилось, см. исправленный §3: у целевой квоты
собственная, независимая от бюджета нумерация приоритетов. Эти два вопроса
(структура API vs сопоставимость priority) независимы — ошибка в одном не
означает ошибку в другом, но и не стоит один вывод переносить на другой.

## 6. Открытые вопросы — то, что нужно решить в формализации метрик

1. **Правило схлопывания узкой «бюджетной семьи» (Бюджет + Отдельная квота +
   Особое право, БЕЗ целевой квоты) до уникального человека — РАЗОБРАНО
   ПОИМЁННО, правило выведено из данных.** Все 47 расходящихся случаев (из
   74 728 пар человек-программа) разобраны по отдельности. Нашёлся чёткий
   механизм: **34 из 47** (30 + 15, см. дальше) полностью объясняются
   `participantStatus = "Отозвано поступающим"` на одном из треков — когда
   абитуриент отзывает заявку по одному льготному треку, приоритет там,
   похоже, не пересчитывается синхронно с оставшимся активным треком. Разбивка:
   - **30 случаев**: ровно один трек активен (не отозван), остальные отозваны
     — приоритет активного трека однозначно «правильный».
   - **15 случаев**: отозваны ВСЕ треки — приоритет для них уже не имеет
     значения (человек не конкурент за место ни в одном треке); согласуется с
     решением по п.4 (`participantStatus`) — эти же 15 и так предлагается не
     считать активными заявками.
   - **1 случай** (`idEpgu=1785708`, «Международная программа по бизнесу и
     экономике»): два действительно активных трека с разными приоритетами (10
     и 8), без отзыва — единственный настоящий тай-брейк на всю выборку.
   - (Формально была ещё пара `idEpgu=1583956`, но там среди активных треков
     приоритет и так совпадает — 4=4 — расхождение только с уже отозванным
     треком, то есть это тот же первый случай, не отдельная проблема.)

   **Итоговое правило**: брать `priority` из трека(ов) со статусом, отличным
   от «Отозвано поступающим»; если таких треков несколько и они совпадают —
   брать общее значение; если совпадений нет (только `idEpgu=1785708`) —
   взять минимальный priority как тай-брейк. Реализуется одной функцией
   агрегации, не требует ручного разбора при запуске на новых срезах данных
   (правило общее, не привязано к конкретным людям).
2. **Целевая квота в метриках программы — РЕШЕНИЕ ПОЛЬЗОВАТЕЛЯ (2026-07-08),
   не открытый вопрос.** Её `priority` не сопоставим с узкой бюджетной семьёй
   (совпадение всего 10.4%, см. §3) — реальные бюджетные места, но с другой
   аудиторией и другим механизмом отбора (целевой приём по договору с
   заказчиком, а не общий конкурс). Пользователь явно решил: **целевая квота
   не входит в M4 (интерес к программе) и не смешивается с общим спросом**,
   рассматривается отдельной диагностической колонкой — по аналогии с тем, как
   «С оплатой обучения» отделена от бюджетной семьи. В `n_applications`/
   `n_unique_applicants` по программе в целом её, вероятно, стоит включать
   отдельной колонкой (не суммировать молча с бюджетом) — это техническая
   реализация уже принятого решения, а не отдельная развилка.
   **Важно для формализующей сессии**: раз целевая квота — это реальные
   бюджетные места, исключение её из M4 не бесплатно — оно систематически
   занижает измеренный спрос для программ с большой целевой квотой (там, где
   много мест разыграно по целевому набору, а не по общему конкурсу).
   Стоит явно посчитать масштаб этого эффекта (например: доля целевых мест от
   общего бюджетного набора по каждой программе, и насколько сильно программы
   с высокой такой долей отличаются от прочих по объёму обычного бюджетного
   спроса) — не для того, чтобы отменить решение, а чтобы дашборд явно
   показывал этот пробел, а не скрывал его молчаливым исключением колонки.
3. **Аналог `K_SERIOUS` — РЕШЕНИЕ ПОЛЬЗОВАТЕЛЯ: `K_SERIOUS = 5`** (не 2, как в
   магистратуре). Обоснование выбора: распределение priority на схлопнутой
   узкой бюджетной семье (74 728 уникальных пар человек-программа) спадает
   плавно, без естественного излома — при `priority ≤ 2` (как в магистратуре)
   «серьёзными» оказались бы 47.7% всех заявок, что для бакалавриата слишком
   мягкий порог (в бакалавриате абитуриенты в среднем подают больше заявлений,
   чем в магистратуре, поэтому шкала растянутее). При `priority ≤ 5` доля
   накопительно — 84.6%. Формализующей сессии нужно просто использовать
   `K_SERIOUS = 5` как константу в M3/M6 (`n_serious`, `demand_pressure`),
   без дальнейшего обсуждения порога.
4. **`participantStatus`** — какие значения считать «активной» заявкой для
   M1–M3. Пользователь не располагает документацией по семантике статусов
   приёмной комиссии и предложил принимать их «as is» — но у этой фразы два
   разных прочтения, и выбор между ними влияет на цифры, а не является чистой
   формальностью. Полное распределение по всем 128 743 заявкам (собрано и
   выгружено в этой сессии):

   | `participantStatus` | штук | доля |
   |---|---:|---:|
   | На рассмотрении | 85 742 | 66.60% |
   | Участвует в конкурсе | 31 776 | 24.68% |
   | Отозвано поступающим | 8 147 | 6.33% |
   | Новое | 2 776 | 2.16% |
   | Ожидание результатов ВИ | 200 | 0.16% |
   | Отклонено вузом | 102 | 0.08% |

   **Вариант А** («as is» буквально): все статусы участвуют в M1–M3 наравне.
   **Вариант Б**: в M1–M3 (которые про актуальный конкурс, а не про
   исторический факт подачи) явно неактивные по смыслу названия статусы —
   «Отозвано поступающим» и «Отклонено вузом» (вместе 6.41% всех заявок) — не
   считаются действующим конкурентом за место, хотя строка остаётся в сыром
   датасете. Это не требует знания бизнес-процесса приёмной комиссии — только
   здравого смысла в названиях статусов, тех же названий, что доступны и
   пользователю. **Рекомендация**: вариант Б как дефолт (по аналогии с тем,
   как в магистратуре M9 — диагностика шума — считалась отдельно от основных
   метрик, а не смешивалась с ними), с явной пометкой, что это первое
   приближение по смыслу названия, а не подтверждённая бизнес-логика, и его
   можно пересмотреть, если найдётся документация. Молчаливое принятие
   варианта А (все статусы наравне) без явного решения — риск: M1–M3 будут
   завышены на ~6.4% (по-разному на разных программах) без объяснения этого
   расхождения где-либо в дашборде.
5. **Источник КЦП (мест приёма) — НАЙДЕН, не открытый вопрос.** В `docs/`
   лежат 4 RTF-файла (`moscow_supply.rtf`, `spb_supply.rtf`,
   `nnov_supply.rtf`, `perm_supply.rtf`, по одному на филиал — совпадает со
   структурой каталога). Одна таблица на программу сразу даёт разбивку по
   всем нужным трекам: колонки — название программы (со ссылкой `<a href>` на
   страницу программы, join-ключ, как в магистратуре), онлайн-признак,
   **Бюджетные места (КЦП)**, **Особое право** (из бюджета), **Целевая квота**
   (из бюджета — внутри ячейки встречаются сноски-ссылки вида `12 <a>1</a>`,
   нужно отделять число от маркера сноски перед парсингом), **Отдельная
   квота** (из бюджета), **Платные места**, **Иностранцы (платно)**.
   Парсер нужно написать по образцу `match_supply_demand.py` из магистратуры
   (`textutil rtf→html`, разбор `<tr>` где первая ячейка содержит `<a href>`,
   агрегатные строки-заголовки направлений без ссылки — отсекать так же, как
   там). До формализации метрик стоит фактически собрать эти 4 файла в единую
   `supply_demand_bak.csv` (программа × филиал × 6 числовых колонок КЦП) —
   это отдельная, не сложная задача парсинга, отделённая от вопроса формул
   M5/M6.
6. **`sumEntranceTestScore` vs `sumCompetitiveScore` vs `sumCompetitiveScoreTarget`
   — РАЗОБРАНО, не открытый вопрос.** Проверено на всех 128 743 заявках без
   исключений: `sumCompetitiveScore = sumEntranceTestScore + achievementsSum`
   — всегда, это стандартный итоговый конкурсный балл, использовать его как
   аналог `total_score` из магистратуры. `sumCompetitiveScoreTarget =
   sumEntranceTestScore + achivementsSumTarget` — отдельное поле, отличается
   от обычного только в целевой квоте (иная шкала учёта индивидуальных
   достижений для целевого приёма), расхождение крайне редкое (4 случая из
   128 743 в этой сессии). Раз целевая квота и так выведена из M4/приоритетных
   метрик отдельной колонкой (см. п.2), для неё можно использовать
   `sumCompetitiveScoreTarget`, для всех остальных треков — `sumCompetitiveScore`.
   **Отдельное наблюдение, важное по срокам**: официальный ранг
   `indexNumberInCompList` пуст у 100% заявок на момент сбора (2026-07-08) —
   конкурсный список ещё не финализирован вузом на этом этапе кампании,
   сверить формулу балла с официальным рангом не удалось. Дашборд,
   показывающий «текущий ранг», должен либо считать его сам по
   `sumCompetitiveScore`, либо явно обозначать, что это неофициальная оценка.
7. **Программы-«близнецы» — ПРОВЕРЕНО, есть, механизм понятен.** Найдено 10
   настоящих близнецов (одинаковое `educationProgram` **и** `filial`, разный
   `educationProgramId`) — например, «Международная программа по бизнесу и
   экономике» в Перми. Во всех 10 случаях один и тот же механизм: под одним
   маркетинговым названием скрываются разные `trainingDirection` (направления
   подготовки/специализации, например «Менеджмент» и «Экономика» под одной
   вывеской) — не дефект данных, осознанная структура набора. Принцип
   переносится из магистратуры без изменений: **связка всегда по
   `educationProgramId`, никогда по отображаемому имени программы**. В файлах
   сбора (`raw/*.json`) это уже так — `educationProgramId` присутствует в
   `meta` каждой записи. Единственное практическое следствие — для парсера
   КЦП (п.5): судя по структуре RTF (заголовки «НАПРАВЛЕНИЕ ПОДГОТОВКИ» над
   группами строк), близнецы там, скорее всего, разнесены по разным строкам
   под разными направлениями — сопоставлять с `raw/` нужно по названию
   программы **и** направлению подготовки вместе, не по одному названию.
