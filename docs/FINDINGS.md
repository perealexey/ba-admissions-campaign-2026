# Находки по структуре данных БАК (pk.hse.ru) — справочник

Документ фиксирует то, что выяснилось на реальных данных после прогонов
`collector_bak.py` (566 конкурсных групп в каталоге; финальный датасет —
405 обычных групп + 32 уникальных пула целевой квоты, иностранцы исключены
по решению, см. §3). Не хроника процесса — только выводы, важные для
дальнейшего переноса слоя метрик (аналог M1–M9 из параллельного проекта
магистратуры, `summer_campaign_26`).

---

## 1. `placeTypeName` — пять регулярных категорий + целевая квота

На срезе от 2026-07-08:

| placeTypeName | групп | заявок |
|---|---:|---:|
| Бюджетные места | 94 | 72 548 |
| С оплатой обучения | 123 | 44 881 |
| Отдельная квота | 94 | 7 495 |
| Квота приёма лиц, имеющих особое право | 94 | 3 388 |
| Иностранцы — отдельный конкурс (исключена из сбора, см. §3) | 86 | 511 |
| Целевая квота (32 уникальных пула после дедупликации, см. §4) | 75 строк каталога → 32 пула | 431 |

«Бюджетные места», «Отдельная квота» и «Особое право» идут группами ровно по
94 — это одна и та же связка мест: у каждой программы с бюджетным набором
автоматически заведены все три квотных трека параллельно.

**В отличие от магистратуры** (`priem47.hse.ru`, где `place_type` был бинарным
budget/commercial), здесь пять независимых по названию категорий — прямой
перенос логики «budget/commercial раздельно» не работает без предварительной
группировки (см. §2).

## 2. Приоритет НЕ независим по каждому `placeTypeName` — кроме целевой квоты

Ключевая находка, проверенная статистически (см. §3): при сравнении `priority`
одного и того же человека (`idEpgu`) на одну и ту же программу
(join по `educationProgramId`, не по названию — устойчиво к программам-тёзкам)
через разные `placeTypeName`:

| Пара `placeTypeName` | Людей с записями в обеих | Priority совпадает |
|---|---:|---:|
| Бюджетные места ↔ Отдельная квота | 5 741 | **99.6 %** |
| Бюджетные места ↔ Особое право | 2 454 | **99.3 %** |
| Бюджетные места ↔ Целевая квота | 260 | **10.4 %** |
| Бюджетные места ↔ С оплатой обучения | 27 252 | 61.2 % |
| Бюджетные места ↔ Иностранцы (историческая цифра, до исключения) | 144 | 25.7 % |

**Интерпретация**: «Бюджетные места», «Отдельная квота» и «Особое право» — не
три независимые конкурсные гонки, а один и тот же приоритет абитуриента на
программу, продублированный в льготных треках (претендует на одно и то же
место программы одновременно по общему конкурсу и по льготе). Это — единая
**«бюджетная семья»** с общей нумерацией приоритетов, но **в неё не входит
целевая квота**.

«С оплатой обучения» и «Целевая квота» — отдельные, независимо пронумерованные
треки. У «С оплатой обучения» 61% совпадений — фоновый шум от похожего
ранжирования, не структурная связь; то же верно и для целевой квоты (10.4%
даже ниже, чем шумовой уровень «С оплатой обучения» — никакой общей нумерации
с бюджетом нет).

**Это исправление ранее сделанного здесь вывода.** Промежуточная гипотеза (по
примеру `idEpgu=1510248`, где бюджетные приоритеты начинались с 2, а не с 1 —
предполагалось, что приоритет 1 отдан целевой квоте) была высказана ДО сбора
целевой квоты и никогда не проверялась на реальных данных. При написании
предыдущей версии этого раздела она была объявлена «подтверждённой» без
пересчёта — это было ошибкой. Пересчёт на собранных 32 пулах целевой квоты
показал обратное: реальное совпадение — 10.4%, то есть целевая квота ведёт
свою отдельную нумерацию приоритетов, не связанную с бюджетной семьёй.

**Следствие для метрик**: при переносе M1–M9 к уникальным людям на программу
нужно сводить бюджетную семью в узком составе — **Бюджетные места + Отдельная
квота + Особое право** (не построчно — иначе один человек считается 2–3 раза).
Целевую квоту нельзя молча объединять с этой семьёй по `priority` — как её
учитывать в общих метриках программы (отдельной колонкой? суммировать только
`n_applications`/`n_unique_applicants`, не полагаясь на сопоставимость
приоритетов?) — открытый вопрос, см. §6. «С оплатой обучения» — отдельная
колонка, аналог `commercial` из магистратуры.

**Масштаб влияния выбора правила агрегации** (для узкой бюджетной семьи из
трёх треков, без целевой квоты): из 74 728 уникальных пар (человек,
программа) 8 025 встречаются в нескольких треках одновременно, и только у
**47** из них (0.6% от восьми тысяч, 0.06% от всех) `priority` реально
расходится между треками. Выбор правила «взять минимальный priority» против
«взять запись с самым продвинутым `participantStatus`» и т.п. меняет
результат только для этих 47 человек — при формализации метрик стоит
рассматривать их поимённо, а не искать общее методологическое решение
вслепую.

## 3. Иностранцы — отдельный конкурс: решение исключить

Проверено по полному каталогу (`cowork_output/catalog_bak.json`, 86 групп) —
**по управляемому полю `placeTypeCode`/`placeTypeId`, не по отображаемому
названию группы** (название — свободный текст, вручную заводится в системе
приёмной комиссии, теоретически может содержать опечатку; код места —
контролируемый справочник, надёжнее). Результат: `placeTypeName` и
`(placeTypeId, placeTypeCode)` находятся в строгом взаимно однозначном
соответствии по всему каталогу (566 строк, без исключений), в частности:

| placeTypeName | placeTypeId | placeTypeCode |
|---|---|---|
| Бюджетные места | `c9a3d888-...` | `Б` |
| Иностранцы — отдельный конкурс | `c9a3d885-...` | `К` |
| С оплатой обучения | `c9a3d887-...` | `К` |

Все 86 записей «Иностранцы — отдельный конкурс» имеют один и тот же
`placeTypeId` = `c9a3d885-148e-11f0-b937-0050560b0254` и код `К` — тот же
буквенный код, что и у «С оплатой обучения» (оба относятся к семейству
«коммерция», просто разные подвиды по гражданству), но другой `placeTypeId`
(отдельное значение справочника). Ни у одной записи нет кода `Б`. Бюджетных
мест для иностранных граждан в этом источнике не предусмотрено — категория
целиком платная, отдельный подвид коммерции по гражданству, а не претендент
на бюджетные места.

**Решение (2026-07-08)**: категория исключена из визуализации и из самого
процесса сбора данных — `collector_bak.py` больше не скачивает группы с
`placeTypeName == "Иностранцы - отдельный конкурс"` (см. `SKIP_FOREIGNERS` в
скрипте, реализовано по аналогии с `SKIP_TARGET_QUOTA`). Уже собранные 86
групп/511 заявок удалены из `bak_data/raw/`.

**Почему это безопасно**: исключение не искажает статистику по бюджетным
местам (см. проверку выше — они физически не пересекаются с бюджетным
набором), только убирает из датасета отдельный платный трек для иностранных
абитуриентов, который не входит в фокус задачи (статистика бюджетной приёмной
кампании).

## 4. Целевая квота: пробел закрыт — дедупликация по паре параметров

Изначальное предположение (что у каждой из 75 групп-заказчиков должен быть
свой различающий параметр запроса) не подтвердилось. Разбор HAR
`har/tselevoi_math_pk.hse.ru.har` (открытая страница конкретной группы
целевой квоты, 2026-07-08) показал обратное:

- Эндпоинт `/applicant?...&setOfCompetitiveGroupId=X&placeType=ЦД&...`
  возвращает **один общий пул абитуриентов**, одинаковый независимо от того,
  через какой из нескольких `competitiveGroupId` (по одному на заказчика)
  зашли. В самих записях абитуриентов `customerIdRabotaVRossii` /
  `customerOfferNumber` всегда `null` — привязки конкретного заявителя к
  конкретному заказчику в этом эндпоинте нет вообще.
- Привязка к заказчику живёт в **отдельном** эндпоинте
  `/quota?setOfCompetitiveGroupId=X&placeTypeId=ЦД` — список организаций
  (`customer`, `contractsCountPerOffer`, свой `competitiveGroupId` на каждую)
  описывает **предложение** (квоты/места по заказчикам), а не заявителей.
- Из 75 строк в каталоге уникальных пар `(setOfCompetitiveGroupId,
  placeTypeId)` — только **32**. Это и есть реальное число различных
  конкурсных пулов; остальные строки — дубли одного и того же пула на разных
  заказчиков.

**Решение**: `collector_bak.py` теперь собирает целевую квоту по 32
уникальным парам (не по 75 `competitiveGroupId`), на каждый пул сохраняет и
список абитуриентов, и список заказчиков (`bak_data/raw_target_quota/`).
Финальный сбор (2026-07-08): 32/32 пула, 431 заявка, 75 строк заказчиков —
ровно столько, сколько было catalog-дублей, подтверждает корректность
дедупликации.

**Важно не путать два разных вопроса.** То, что описано в этом разделе
(структура API, дедупликация пулов) — установлено надёжно и не пересматривается.
А вот побочное предположение, сделанное по ходу (что целевая квота использует
ту же нумерацию `priority`, что и остальная бюджетная семья) — было
преждевременным и не подтвердилось на данных: реальное совпадение приоритетов
с бюджетом — 10.4%, см. исправленный §2. Дедупликация API-пулов корректна вне
зависимости от этого — правки в §2 не отменяют содержимое этого раздела.

## 5. Программы-«близнецы»: детальный разбор (пул абитуриентов и экзамены)

10 подтверждённых случаев (одно отображаемое имя + филиал, разные
`educationProgramId` и `trainingDirection`, см. §2 практическое следствие и
разбор в handoff-документе §7) распадаются на два явно разных по природе
кластера — не единое явление, а минимум два механизма.

### Пересечение пулов абитуриентов (по `idEpgu`, срез 2026-07-08)

**Высокое пересечение** — заметная доля абитуриентов подаётся на оба
направления одновременно, как на альтернативы:

| Программа | Филиал | Направления | Пересечение (Jaccard) |
|---|---|---|---:|
| Международная программа по бизнесу и экономике | Пермь | Менеджмент ↔ Экономика | 0.74 (90%/81%) |
| Разработка информационных систем для бизнеса | Пермь | Бизнес-информ. ↔ Программная инж. | 0.66 (80%/79%) |
| Компьютерные науки и технологии (ПМИ ↔ Программная инж.) | Н.Новгород | | 0.67 (75%/86%) |
| Международная программа по бизнесу и экономике | Н.Новгород | Менеджмент ↔ Экономика | 0.71 (90%/77%) |
| Международная программа по бизнесу и экономике | СПб | Экономика ↔ Менеджмент | 0.57 (69%/77%) |
| Политология и мировая политика | СПб | Межд. отношения ↔ Политология | 0.52 (64%/73%) |
| Компьютерные науки и технологии (Бизнес-информ. ↔ ПМИ) | Н.Новгород | | 0.40 |

**Низкое пересечение** — практически независимые аудитории:

| Программа | Направления | Пересечение (Jaccard) |
|---|---|---:|
| Разработка игр и цифровых продуктов | Программная инженерия ↔ Дизайн | 0.015 (1.9%/6.7%) |
| Программа двух дипломов ВШЭ–Кёнхи | Публ. политика ↔ Заруб. регионоведение | 0.11 (17%/23%) |
| Реклама и связи с общественностью | Реклама и СО ↔ Медиакоммуникации | 0.11 (15%/28%) |
| Межд. программа «Межд. отношения и глоб. иссл.» | Межд. отношения ↔ Публ. политика | 0.13 (21%/24%) |

### Причинный механизм: совпадение требуемых экзаменов ЕГЭ

Сверка `applicantEntranceTests` (список требуемых вступительных испытаний) по
`educationProgramId` каждого направления показала чёткую корреляцию,
объясняющую два кластера выше:

- **Высокое пересечение → экзамены совпадают или почти совпадают.** Например,
  МПБЭ (Н.Новгород, Пермь): у «Менеджмент» и «Экономика» **идентичный** набор
  экзаменов — абитуриент подаёт один и тот же комплект баллов на оба
  направления без дополнительных условий, отсюда и высокое пересечение пулов.
  КНТ «ПМИ»/«Программная инженерия» (Н.Новгород) — тоже идентичны
  (Информатика, Математика, Русский, Физика).
- **Низкое пересечение → экзамены различаются существенно.** Самый яркий
  случай — «Разработка игр и цифровых продуктов»: направление «Программная
  инженерия» требует Информатику/Математику/Физику (STEM-профиль), направление
  «Дизайн» — Литературу вместо этого набора (гуманитарно-творческий профиль).
  Это два физически несовместимых комплекта ЕГЭ — один и тот же человек в
  подавляющем большинстве случаев не может одновременно располагать баллами
  по обоим наборам, что напрямую объясняет почти нулевое пересечение
  (1.9%/6.7%), а не совпадение. Похожая вилка История/Обществознание видна у
  «Программы двух дипломов ВШЭ–Кёнхи» и у «Межд. программы "Межд. отношения и
  глоб. иссл."».

**Вывод**: набор требуемых экзаменов — по всей видимости, причинный механизм,
а не просто коррелирующий факт. Совпадающие экзамены делают переключение
между направлениями «бесплатным» для абитуриента (высокое пересечение);
разные экзамены физически разделяют аудитории по профилю подготовки
(низкое пересечение), независимо от общего маркетингового названия.

**Следствие для метрик/дашборда**: близнецов с высоким пересечением
(совпадающие экзамены), возможно, разумнее объединять в отчётности как один
конкурс с двумя специализациями — раздельный показ там создаёт видимость
двух независимых программ там, где реально один пул выбирает между двумя
опциями. Близнецов с низким пересечением (разные экзамены) объединять
нельзя — это исказит картину, раздельный показ (уже сделанный, см. §2)
оправдан. Решение по объединению/разделению в отчётности не принято —
это открытый вопрос на будущее, не пересмотр текущей реализации дашборда.

## 6. Открытые вопросы (не закрыты этим документом)

- **Как включать целевую квоту в метрики программы**, если её `priority` не
  сопоставим с бюджетной семьёй (см. §2)? Варианты для обсуждения при
  формализации метрик: отдельная колонка (как «С оплатой обучения»);
  суммировать только объёмные показатели (`n_applications`,
  `n_unique_applicants`) без опоры на priority; не включать в M4
  (желанность) вовсе, раз приоритет не сравним с остальными. Решения нет.
- Не проверено, участвует ли «Иностранцы — отдельный конкурс» на самом деле в
  какой-либо общей нумерации приоритетов с «С оплатой обучения» для того же
  человека на РАЗНЫЕ программы — сейчас это не важно, так как категория
  исключена из датасета, но если решение по иностранцам изменится в будущем,
  этот вопрос придётся пересмотреть.
- Источник КЦП (мест приёма) по каждой программе/треку для сопоставления
  спрос/предложение (аналог `match_supply_demand.py` из магистратуры) — пока
  не исследован для бакалавриата. Для целевой квоты частично закрывается полем
  `contractsCountPerOffer` из `/quota` (см. §4) — для остальных треков
  источник не найден.
